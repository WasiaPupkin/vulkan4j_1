# Java Project Standards & Methodology

Follow these rules to navigate and analyze the Java codebase efficiently:

1. **Direct Communication**: If the project structure is ambiguous and RAG results are conflicting, do not guess. List the files you found and ask the user for the correct entry point.
2. **Verification**: After making a change or concluding an analysis, summarize *what* was discovered and *why* it resolves the task.
3. **Package-First Navigation**: 
   - Always prioritize the `src/main/java` structure. 
   - Before reading class content, identify the package via `list_files` to understand the domain context.

4. **Interface & Implementation Logic**:
   - If you encounter an `interface`, do not get stuck reading it multiple times. Immediately search for implementing classes (e.g., `implements [InterfaceName]` or files in `impl` packages).
   - Use `search_files` to find the concrete logic instead of guessing based on the interface.

5. **Dependency Analysis (Maven/Gradle)**:
   - If a symbol or library is unknown, check `pom.xml` (Maven) or `build.gradle` (Gradle) exactly once to identify the version and library. Do not re-read build files unless dependencies change.

6. **Spring/Framework Awareness**:
   - If this is a Spring Boot project, search for `@Component`, `@Service`, or `@Repository` annotations to find the entry points of business logic.
   - When looking for API endpoints, start by searching for `@RestController` or `@RequestMapping`.

7. **Lombok & Generated Code**:
   - Be aware that methods like `getters`, `setters`, and `builders` might be generated by Lombok (`@Data`, `@Getter`, etc.). If you don't see a method in the code, check the class annotations before searching further.

8. **Error Handling**:
   - Check `GlobalExceptionHandler` or classes annotated with `@ControllerAdvice` to understand how the project handles errors, instead of tracing every `try-catch` block manually.
